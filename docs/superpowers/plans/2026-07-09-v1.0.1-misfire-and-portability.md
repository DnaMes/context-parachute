# context-parachute v1.0.1 Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Fix the confirmed 1M-context misfire (watcher fires at ~16% on Opus-1M sessions) via an honest `context_window` override, and make the watcher run on macOS/BSD by removing `tac`.

**Architecture:** No trigger-logic change. B2 is documentation + an optional stderr warning; the `context_window` config knob already exists and is already wired into the watcher. C2 swaps one `tac`-based pipeline for a portable `tail`-based one that produces byte-identical output (verified against all fixtures).

**Tech Stack:** bash, jq, shellcheck. Plain-bash test runner (`tests/run-tests.sh`), no bats.

## Global Constraints

- Hooks are **fail-open**: any internal error → `exit 0`, never blocks the session; diagnostics to stderr with a `WARN:` prefix, never swallowed silently.
- Pure `bash` + `jq`, `shellcheck -S warning` clean. No new runtime dependencies.
- POSIX-portable shell: no `tac`, no `tail -r`, no GNU-only flags in shipped hooks.
- Artifacts and copy in **English**.
- Version is the `VERSION` file at repo root (single source of truth). The test suite asserts `VERSION == top CHANGELOG entry == latest git tag`.
- Config precedence: built-in defaults → `~/.claude/parachute.json` → `.parachute/config.json` (cwd). Invalid config → `WARN:` to stderr + fall back to defaults.

### Task 1: Portable reverse-read (C2) — replace `tac`

**Files:**
- Modify: `hooks/parachute-watch.sh:77-79`
- Test: `tests/run-tests.sh` (existing "Token-sum correctness" section already covers behaviour; add one portability assertion)

**Interfaces:**
- Consumes: nothing new.
- Produces: no interface change. `USAGE_LINE` holds the same value as before — the last main-chain assistant `.message.usage` object from the transcript tail.

**Background:** The current pipeline reverses the file with `tac` (GNU-only) and takes the first match. `tac` is absent on macOS/BSD; `tail -r` (the BSD reverse) is absent on GNU — so a reverse-read needs two code paths. Taking the **last match** of a forward `tail` needs neither. Verified byte-identical on `normal`, `cache-heavy`, `corrupt`, `at-80` fixtures.

- [ ] **Step 1: Confirm the current behaviour is captured by existing tests**

Run: `bash tests/run-tests.sh 2>&1 | grep -A6 "Token-sum correctness"`
Expected: PASS for `normal -> 80%`, `cache-heavy -> 79% silent`, `sidechain line ignored`, `corrupt lines skipped`. These are the behavioural guard for this change.

- [ ] **Step 2: Replace the `tac` pipeline with a portable `tail` pipeline**

In `hooks/parachute-watch.sh`, replace lines 77-79:

```bash
USAGE_LINE="$(tac "$TRANSCRIPT" 2>/dev/null | head -n 500 \
    | jq -c -R 'fromjson? | select(.type=="assistant" and (.isSidechain != true)) | .message.usage' 2>/dev/null \
    | head -n 1 || true)"
```

with:

```bash
# Portable reverse-read: take the LAST matching line from the last 500, instead
# of reversing the file to take the first. Avoids tac (GNU-only) and tail -r
# (BSD-only), so the watcher runs on Linux and macOS alike. Verified to produce
# the identical usage line as the previous tac-based pipeline on all fixtures.
USAGE_LINE="$(tail -n 500 "$TRANSCRIPT" 2>/dev/null \
    | jq -c -R 'fromjson? | select(.type=="assistant" and (.isSidechain != true)) | .message.usage' 2>/dev/null \
    | tail -n 1 || true)"
```

Expected: `34 passed, 0 failed` (same as before the change).

### Task 2: Overflow warning + `context_window` override docs (B2)

**Files:**
- Modify: `hooks/parachute-watch.sh` (add a warn near the percent computation, ~line 87)
- Modify: `README.md` (Configuration section — document the 1M override + example)
- Modify: `config/parachute.default.json` is NOT changed (default stays `200000`); add a note in the README instead
- Test: `tests/run-tests.sh` (two new assertions: override silences a would-fire case; overflow warns)
- Test fixture: `tests/fixtures/at-80.jsonl` is reused (it sums to 160000 = 80% of 200000)

**Interfaces:**
- Consumes: the existing `WINDOW` variable (already loaded from config, `parachute-watch.sh:37,49,52`) and `TOKENS` (computed at `:82-85`).
- Produces: a stderr `WARN:` line when `TOKENS > WINDOW` and `WINDOW` is still the built-in default; the trigger decision is unchanged.

**Background:** Auto-detecting the window was rejected (model string lacks `[1m]`; empirical inference is circular at the 160k threshold; no structural transcript signal). The honest fix is the existing `context_window` knob — users on 1M set it to `1000000`. The warning makes a silent misfire visible.

- [ ] **Step 1: Write the failing test — override silences a would-fire fixture**

`at-80.jsonl` sums to 160000 tokens = 80% of the default 200000 window → fires. With `context_window: 1000000` it is 16% → silent. Add to `tests/run-tests.sh`, in a new section before "Fail-open":

```bash
section "context_window override (B2)"

# at-80 fires at the 200k default; a per-project 1M override must silence it.
ov_dir="$(mktemp -d)"
mkdir -p "${ov_dir}/.parachute"
printf '{"context_window":1000000}\n' > "${ov_dir}/.parachute/config.json"
input="$(jq -nc --arg t "${FIXTURES}/at-80.jsonl" '{transcript_path:$t, session_id:"s-ov"}')"
out="$(cd "$ov_dir" && printf '%s' "$input" | TMPDIR="$RUN_TMP" bash "$WATCH" 2>/dev/null)"
[[ -z "$out" ]] && ok "1M context_window override silences 160k session" || bad "1M override silences (got: ${out:0:50})"
rm -rf "$ov_dir"
```

- [ ] **Step 2: Run it — expect FAIL (override path works, but assertion is new; confirm it actually exercises the override)**

Run: `bash tests/run-tests.sh 2>&1 | grep -i "override"`
Expected: `PASS 1M context_window override silences 160k session` — the config load path already exists (`load_config` reads `context_window`), so this may pass immediately. That is fine: it locks in the behaviour the release documents. If it FAILS, the config path is broken and must be fixed before proceeding.

- [ ] **Step 7: Document the override in the README**

In `README.md`, in the Configuration section, after the config-keys table, add:

```markdown
### 1M-context sessions

context-parachute cannot reliably detect the context-window size from the
transcript (the model string does not carry the `[1m]` suffix, and there is no
structural window field). If you run a 1M-context model (e.g. Opus 4.8 `[1m]`),
set the window explicitly so the threshold measures against the real budget —
otherwise the parachute fires far too early (at ~16% of a 1M window):

```json
{ "context_window": 1000000 }
```

Set it globally in `~/.claude/parachute.json`, or per-project in
`.parachute/config.json`. When observed tokens exceed the configured window,
the watcher prints a one-line `WARN:` to stderr pointing you at this knob.
```
